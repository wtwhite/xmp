mms help v1.5

 

mms is a command line utility that performs the minmax squeeze on phylogenetic alignments to try and produce a maximum lower bound  for that alignment’s parsimony score.

 

1 Installing

 

1.1 installing mms

 

·        Windows 95/98/Me: copy mms.exe to c:\windows\command

·        Windows NT/2000: copy mms.exe to c:\winnt\system32

·        Windows XP: copy mms.exe to c:\windows\system32

 

·        Linux binary: copy mms to /usr/local/bin

·        Linux source: extract archive and cd into its directory

Type “make”

Then “make install”

 

To compile the mms linux source you will need a working c++ development environment including the g++ compiler, to install you may have to be logged in

As root.

  

1.2 notes

 

You may need admin/root privileges to install the program to these locations.

It is not necessary to install mms to the system directory; you can use it from any folder, but if it is not in the system directory you will have to navigate to the folder that mms is in before you will be able to use it. Alternatively on windows you can add the location of the program to the windows path environment variable.

 

2 Running the program

 

2.1 Command line argument syntax

 

All arguments must be entered with an equals sign between it and its corresponding value, and no spaces in between. All argument value pairs must be separated by spaces i.e.

 

mms [argument]=[value] [argument1]=[value1]

 

Example:

mms alignment=test.phy iterations=10 nochanges=50

 

2.2 Required argument:

 

Alignment=[filename]

This argument must specify the filename of a phylip alignment for the program to run the minmax squeeze on. If this argument is not specified, the program will not run.

 

2.3 Optional Arguments:

 

   iterations=[value>0]

This argument specifies how many passes of the mms search algorithm are performed. If this parameter is not specified the program uses the default value of 5

 

   nochanges=[value>0]

This specifies the maximum number of iterations the search algorithm should proceed if the max lower bound has not changed before quitting the pass. The default value if this parameter is not specified is 5

 

   log=[filename]

This specifies a filename to write an ongoing log of the programs progress to. By default this option is turned off.

 

   loadpart=[filename]

This specifies the filename of a partition to load into the program to use as a starting point for the search algorithm.

By default this option is turned off.

NOTE: if you choose to load a partition, then you cannot use the exclude argument. The reason for this is that a partition file keeps a record of what sites were excluded when it was saved (hence it automatically adjusts the alignment upon loading) This is so that it can restart, knowing that the alignment was the same as it was last time.

 

  generatepart=[algorithm]

This specifies an algorithm to use to generate a starting partition for the search algorithm to use. Note: if both loadpart and generatepart arguments are specified, the generatepart argument will be ignored.

[algorithm] can take on the following values. If this argument is not specified then default is used.

 

default: This generates what is known as the trivial partition where every site

in an alignment is a separate part.

 

pairsplit: This generates a starting partition based on the idea of splitting apart

pairs of taxa. The parts generated consist of those sites that split apart the most taxa pairs.

 

likelygroup: Likelygroup is different from the other two algorithms in that it

intelligently readjusts the partition as the search runs, rather than generating a starting partition that remains unchanged for the duration of the search.

It begins with a trivial partition and for x passes of the search algorithm (see learningpasses argument) builds up a table recording how often sites appear with other sites in the same part.

After the learning passes have completed a partition is generated by grouping together the parts that appear together most often. (for specifying what is considered a likely grouping see the groupcutoff argument). Parts which don’t appear often with other parts are added in trivial parts.

This will generate a constantly improving partition that includes sites that appear often together and therefore logically offer a high bound improvement.

 

   groupcutoff=[0<value<1]

This argument specifies what the likelyCutoff considers to be a likely grouping of sites. The value must be a floating point value between 0 and 1 and represents the percentage of times one part appears with another for it to be considered a likely grouping.

e.g. a value of 0.4 would mean that all groupings of sites which occur in 40% or more passes of the search algorithm will be considered likely and will be grouped accordingly. By default this value is set to 0.35. This argument is ignored if the likelygroup algorithm is not used

 

   learningpasses=[0<value<1]

This argument specifies the percentage of the total passes to use for building up a likelygroup table before actually generating likelygroup partitions.

e.g. a value of 0.5 would mean that half of the passes would be used for generating a likelygroup table before actually generating likelygroup partititions in the remainder of the passes. By default this value is set to 0.5. This argument is ignored if the likelygroup algorithm is not used

           

   savepart=[filename]

This argument specifies a filename to save the best partition found by the search. By default this option is turned off.

 

 

   seed=[0<value<1000]

This argument specifies a seed number with which to initialize the random number generator for the search algorithm. If this argument is not specified a pseudo random seed number is generated based on the system time.

 

   upper=[value]

This argument specifies a value for the upper bound on an alignment. If this value is reached during a search, the program will cease execution as this is the value that the program is searching for. If this argument is not specified then the program will continue searching until the full number of passes have been made (see iterations parameter). This argument ovverides any upper value set by the tree parameter

 

   tree=[filename]

This argument specifies a tree file from the alignment. If specified the program uses the bound generated from the tree as the upper parameter and uses the information on the tree to optimize the search.

NOTE: this tree must be in newick format

By default this option is turned off. 

 

Savestartpart=[filename]

This argument specifies a file to save the starting partition  to. This can be useful in the case of likelygroup partitions which take a long time to generate. You can save time by running the program once and saving the starting likelygroup partition, so in future runs you can load the partition in rather than generate it again. By default this option is turned off.

 

Retrivpasses=[value]

         This argument specifies how many times to retrivialise a partition once the bound has stopped improving. The default value for this argument is 1 and will only work when a tree is also specified.

 

Savesitecosts=[filename]

saves a sorted list of site costs from a tree (only works when a tree is specified) by default this option is turned off.

 

Exclude=[exclusion string]

Excludes sites specified in the exclusion string, the string is formatted as a series of integers (or ranges of integers) separated by commas

            e.g. 0,1,3,4-7,10

            Note: first site in an alignment is 0

Note: placing a ‘–‘ character between site numbers will exclude all sites between the values (inclusive of the values specified).

Note: you cannot use this argument when you use the loadPart argument

  

 

3 Some tips and general Info

 

3.1 Merge and oneElement moves

The process of finding a maximum lower bound on an alignment is done by repartitioning sites in the alignment. This process is achieved in this program by two move types, merge and oneElement.

A merge move takes two parts in a partition and merges them into one partition

A oneElement move removes a site from one part and gives it to another part.

Generally merge moves offer greater increases to the bound when the partition is close to trivial. However as the partition contains less and less parts, oneElement moves become more effective.

Merge moves are favoured by the program for as long as possible in the searching process because they are much faster to execute

 

3.3 Retrivialising

Retrivialising is a process which returns parts in the partition to a trivial state (i.e. one site to one part). This technique is used when tree data is loaded into the program. Once there are no more good moves the program checks which parts are not optimal (according to the tree data) and then retrivialises those parts. This gives the program a another chance to repartition those sites to hopefully generate an optimal partition

 

3.4 Partition file format

the generic format is as follows (for this description words inside [] characters

would appear as the appropriate value in a real file)

 

// ********

// some header info

// about the file

// ********

 

[siteExclusionList]

 

[partNum]

 

[partSize]( [siteNum0],[siteNum1])//some info on the bound of this part

[partSize]([siteNum0],[siteNum1]) //some info on the bound of this part

etc.

 

3.5 trees with internal nodes

mms supports loading trees with named internal nodes in the standard newick format

            i.e. (leaf1,leaf2)nodeName

 

3.6 excluding sites

All sites in an alignment whose data is the same across all taxa, (i.e will have a bound of 0 and thus is irrelevant for the minmax squeeze) are automatically ignored by the program.

Other sites must be excluded manually with the exclude argument. A list of site costs can be generated using the savesitecosts argument and this can be useful in helping to choose sites to exclude in future.

 

3.7 site cost file format

the generic format is as follows (for this description words inside [] characters

would appear as the appropriate value in a real file)

 

*********

 some header Info

*********

 

    [siteNum] : [siteCost]

                [siteNum] : [siteCost]

            etc.
